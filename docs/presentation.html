Key Terms

Glossary

Six terms that cover everything on screen for the rest of this talk

SHA-256 hash function
Cryptographic fingerprint of data
One-way, fixed-length (256-bit) digest. Used to fingerprint every chunk of text at ingestion.
Merkle Tree data structure
Binary tree of hashes committing to a document
Each leaf = hash of one chunk. Each parent = hash of its two children. The root commits to all of them, in order.
Merkle Proof sibling path
Proof one chunk belongs to the signed set
The sibling hashes from a leaf up to the root — proves inclusion without re-hashing every other chunk.
Ed25519 signature scheme
Elliptic-curve digital signature (via PyNaCl)
A private signing key produces a signature only its paired public key can check — unforgeable without that key.
Attestation signed claim
Binding of an output to its exact inputs
Binds answer hash + cited chunk hashes + query hash + model, signed by the service key — non-repudiable.
in-toto / DSSE open standard
Supply-chain provenance statement (ITE-6)
A spec-compliant, DSSE-signed export — the bridge to the in-toto / TUF / sigstore ecosystem.

Architecture

Pipeline

The provenance layer wraps standard RAG at three points, untouched otherwise

1
Ingest
SHA-256 each chunk, build the per-document Merkle tree, sign the root with the publisher key.
2
Retrieve
Embed the query, take top-N chunks, then re-hash + verify Merkle path + verify root signature for each.
3
Generate
LLM answers with inline citations; bind chunk hashes + answer hash + query hash, sign with the service key.
4
Verify
A standalone process re-runs the same checks offline, using only the two public keys.

Trust Model

Two Keys

Two independent keys keep two questions separately answerable

  • The publisher key signs the Merkle root of each document at ingestion — attests to the origin and integrity of sources.
  • The service key signs each answer attestation at generation — attests to what the system did with those sources.
  • Separating them means "was a source altered?" and "was an answer forged?" are answerable independently.
  • Compromise of one key does not silently implicate the other.
  • Public keys (.vk) are committed as the trust anchor; private keys (.sk) are gitignored, never committed.
The trust anchor lives deliberately outside the vector store an attacker would tamper. roots.json · attestation_log.jsonl · keys/*.vk

Threat Model

What It Doesn't Claim

It proves the provenance of an answer, not its truth — stating that boundary is deliberate

Trust boundaries

Malicious or negligent publisher

A publisher can sign false content. The signature makes them accountable — non-repudiable — but doesn't adjudicate truth. Provenance ≠ correctness.

Key compromise, rotation, revocation

Theft of either private key breaks its guarantee. A rotated key invalidates prior attestations with no transparency log to reconcile them yet.

Freshness / rollback

Each ingestion is an independently signed snapshot. An older, authentically signed snapshot could still be replayed — no TUF-style monotonic timestamp yet.

Explicit non-goals

Retrieval-selection attacks

An adversary who influences the query or embedding space can surface authentic but misleading chunks. Integrity doesn't imply honest selection.

Prompt injection

Instructions smuggled through retrieved content can subvert the answer while every integrity check still passes. A separate problem, deliberately out of scope here.

Confidentiality & availability

No protection of source secrecy, no resistance to denial of service. The asset protected is integrity of the chain, not secrecy or uptime.

Real-World Stakes

Applications

Where this pattern matters, and what it actually stops

Applications

Regulated research & compliance

Legal, financial, and medical domains where an AI-generated answer must be auditable after the fact — provably traceable to a specific, unaltered source, not just plausible-sounding.

Enterprise knowledge bases

Internal wikis, incident postmortems, and policy Q&A, where accountability for what the system told an employee matters as much as the answer itself.

Scientific literature assistants

This demo's own domain: research tools over papers, where an answer must prove it traces to real, unaltered papers — not a hallucinated citation.

Vulnerabilities Stopped

Data poisoning / silent corpus tampering

Editing or swapping indexed content after the fact to manipulate future answers — caught by the read-hook hash + Merkle proof + root signature check.

Citation fabrication

An answer claims support from a source it never actually used — caught because the attestation binds the answer to the exact chunk hashes placed in context.

Post-hoc output tampering

An answer altered after generation — in a log, a cache, a compliance record — to misrepresent what the system actually said. Caught by the signed answer_sha256.

Signed at the Source,
Signed at the Answer

Independently checkable end to end — ask a question, tamper a source, watch the citation flip, verify it from a separate process holding only the public keys.
